# Работа с Git и GitHub

# 1. Проверка наличия установленного Git 
В терминале выбрать комманду `git --version`.
Если Git установлен, появится сообщение с информацией о версии программы. Иначе будет сообщение об ошибки.
  

# 2. Установка Git
Загружаем последнюю версию файла с сайта https://git-scm.com/downloads.
Устанавлием с настройками по умолчанию.


# 3. Настройка Git 
При первом использовании Git необходимо представиться. Для этого нужно ввести в терминале 2 комманды: 
```
git config --global user.name «Ваше имя английскими буквами»
git config --global user.email ваша почта@example.com
```


# 4. Инициализация репозитория
Инициализация или создание репозитория производится командой `git init`.
Команду нужно ввести 1 раз. После этото появится сообщение о том что инициализация выполнена.


# 5. Запись изменений в репозиторий
После редактирования файла можно ввести команду `Git status` для того чтобы проверить состояния файлов в рабочей директории и индексе: какие файлы изменены, но не добавлены в индекс; какие ожидают сохранения в индексе. Не сохранённый, редактируемый файл отображается красным цветом. 

Выполнив команду `git diff` вы увидете разницу между вашим текущим рабочим каталогом и вашей промежуточной областью.
 Зеленым показаны строчки, которые были добавлены, а красным которые удалены. 

Для подготовки к сохранений изменений в проекте используется команда `git add .`. 
Выполнив эту команду, вы подготавливаете конкретные изменения к фиксированию в репозитории. Эта команда имеет накопительный эффект, `git add` может быть использована многократно перед тем, как будет создан коммит.  

После этого вы можете повторно использовать команду `Git status` чтобы посмотреть файлы подготовленные к сохранению. Подготовленный файл будет отображаться зелёным цветом.

Затем вы вызываете команду `git commit -m" после m можно записать какие именно изменения внесены в файл который вы будете сохранять,(например: git commit -m"Добавили заголок Работа с Git" ) обязательно использовать кавычки как показано тут"`. После того как записали нажмите enter, чтобы сохранить состояние изменений в историю проекта. 
Таким образом изменения в проекте регистрируются только после выполнения команды `git commit`. 

После сохранения (коммита) в терминале появится информация об успешном изменение файла (пример : git commit -m "добавили подраздел 5 - Запись изменений в репозиторий"
[master 9274431] добавили подраздел 5 - Запись изменений в репозиторий
 1 file changed, 15 insertions(+), 1 deletion(-))
 Так же вы можете ввести команду `Git status` для того чтоб увидеть информацию о сохранении.

 Так же есть ещё одна очень удобная команда для фиксации изменений и сохранения комитов без использования отдельной функции `Git add`, если ввести такой коммит с параметрами -a, так он будет выглядеть на примере - `git commit -a -m"(комментарий что именно изменили)"` эта команда является комбинированной она выполняет подготовку к изменению, а потом сохранения(фиксацию),тоесть одной командой выполняется 2 команды последовательно. 
 С помошью этой комбинированной команды нельзя добавлять новые файлы в отслежевание, можно фиксировать только изменения файлов которые уже есть, она подготавливает к следующему коммиту все изменеия которые есть в репозитории,
 к примеру если у нас изменения в нескольких файлах, данная команда `git commit -a -m"(комментарий что именно изменили)"` добавит все измения и запишет в один коммит. 


 # 6. Просмотр истории всех коммитов
Командa `Git log` поможет посмотреть историю всех сохранений(коммитов) которые мы создали.
Выполнив эту команду, увидем список всех коммитов с хэш кодами. 
Хеш код коммита  — это специальная метка, позволяющая отличать одни коммиты от других, можно сказать что это разные версии.

Пример хеша коммита: d1cw4e2bd74e03ajj732assb476392ff77f24e47 
Под хеш кодом указано: автор(создатель коммита), так же дата и время создания и комментарий с которым был сохранён коммит.
`Git log` многофункциональная команда, возможные варианты использования можно посмотреть при помощи дополнительного параметра `-h`, написав -h вызываем помощь это help, информация по команде, пример `Git log -h`, после этого вы увидете дополнительные варианты использования этой командой.

Так же можно вывести список `Git log` в сокращённом виде, для этого введите команду `Git log --oneline` это значит в одну строку, сокращённый вариант вывода.
В этом случае Git выдаёт нам первые 7 символов хэш кода, а так же комментарии к фиксации.


# 7. Перемещение между коммитами (сохранениями)
Для того чтобы вернуться к предыдущей версии нашего файла(коммита), нужно вызвать список коммитов командой `Git log --oneline` или полной версией `Git log` , найти интересующую версию к которой хотим вернуться,скопировать хэш код. Не обязательно копировать все символы коммита, главное, скопировать не меньше 4-х символов начиная с начала, далее используем команду `Git checkout после этого слова добавляем сюда скопированные символы коммита"`. Пример: Git checkout d1cw. Выполняем эту команду и попадаем на прошлую версию нашего сохранённого файла.
Для того чтобы вернутся обратно в актуальное состояние к нашей последней версии коммита в которой мы работали, нужно ввести команду - `Git checkout master` или `Git switch -`.

# 8. Игнорирование файлов
Для того, чтобы исключить из отслеживания в репозитории опрелённые файлы и папки необходимо создать там файл ***.gitignore*** и записать в него их названия или шаблоны,соотвествующие таким файлам или папкам.
Для создания шаблона для игнорирования всех файлов одного типа нужно ввести в файле ****.(формат файла без скобок)*** пример: ****.png***.

# 9. Создание веток в Git
В Git ветки — это элемент повседневного процесса разработки. По сути ветки в Git представляют собой указатель на снимок изменений. Если нужно добавить новую возможность или исправить ошибку (незначительную или серьезную), вы создаете новую ветку, в которой будут размещаться эти изменения.

 По умолчанию имя основной ветки в Git - **master**. Создать ветку можно командой: 
 ``` 
 git branch <имя новой ветки>
 ```
 Список веток в репозитории можно посмотреть с помощью команды 
 ```
git branch
 ```
 Текущая ветка будет отмечена звёздочкой: **master**

Для переключения между ветками используется команда:
```
git checkout <имя ветки на которую хотим переместиться>
```
Так же есть удобные команды для создания с последующим перемещением на только что созданную ветку 
```
git checkout -b <название новой ветки> 
```
```
git switch -c <название новой ветки>
```
# 10. Слияние веток и разрешение конфликтов Git
Для того чтобы выполнить команду слияния нужно перейти на главную ветку **master**, после перехода ввести команду 
```
git merge <вводим название ветки которую нужно слить из нашего черновика в master>
```
После того как произошло слияние вся информация из ветки появится в главной ветке **master** так же в терминале появится статистика с обновлённым коммитом стратегия слияния - fast forward. Далее можно ввести команду ***git log --graph*** в терминале появится список коммитов, слева показана пунктирной линией стратегия слияния **fast forward** отображение линейной ветки, без лишних коммитов. Из двух веток произошло слияние в одну. 

Если вы рабоете в созданной ветке, сохранили изменения в ней. После этого перешли в главную ветку **master** записали там ту же информацию и сохранили, но добавили какие то изменения в главную ветку и ваши ветки отличаются информацией, при слияние веток командой -
```
Git merge <название ветки которую вы сливаете с веткой master>
```
произойдёт конфликт автослияния.
Появится окно с подсветкой от visual studio code.
Пример: 

![пример](Git.png)

Для разрешения конфликта необходимо выбрать из предложенных вариантов.

Принять текущее изменение - сохранить информацию из ветки Git master.

Принять входящее изменение - сохранить информацию из ветки черновика.

Принять оба изменения - сохранятся оба варианта, которые можно отредактировать.

Сравнить изменения - сравниваем оба варианта, смотрим отличия .

После этого нужно обязательно выполнить commit слияние, для этого нужно ввести новый 
```
git commit -a -m"разрешили конфликт"
```
# 11. Удаление слитых веток
После успешного слияния веток, ветку которую мы использовали в качестве черновика, нужно удалить за ненадобностью.
Для удаления ветки воспользуемся коммандой 
```
git branch -d <указываем имя ветки которую хотим удалить>
```
Выполняем команду и видим в терминале сообщение, об успешном удаление ветки.

Есть некоторые ограничения при удалении веток.

Например: нельзя удалить ветку находясь внутри этой ветки.
Так же если ветка которую вы хотите удалить ещё не прошла слияние с осоновной веткой **master**, удалить её просто не получится, при удаление git выдаст ошибку предупреждая, о том что ветка не слита.

Если вы всё таки хотите удалить принудительно удалить не слитую ветку, нужно ввести команду:
```
git branch -D <указываем имя ветки которую хотим удалить>
```
Для восстановления веток можно использовать историю commitов и мы всегда сможем вернутся к удалённой ветки.
